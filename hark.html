Harmonic Relational Key Memory

Really, this project was a sub-project of my PhD, and was the result of a merging between several originally distinct thoughts that were running through my mind.

First, is it possible for vague ideas to become precise ones? The best analogy I can think of for what it even MEANS for an idea to be vague, is to say something like it's "blurry"... but that can't be right, because that would imply that there was somehow an existing "crisp" idea in our mind that was BEING blurred, which would't make any sense. Somehow, the representational substrate must support representations that have an INTRINSICALLY variable crispness.

Second, I'm interested in how information can be quickly stored, deleted, and modified in neural systems to support rapid learning. Because of that, I quickly gravitated towards different forms of Hebbian learning, in particular, the form of learning used in heteroassociative memory systems. You can think of such memory systems as being sort of like a dictionary, mapping "key" vectors to "value" vectors. But, I wondered if there was some way that the keys could be systematically organized to represent an underlying, lower-dimensional continuous space. We have strong evidence that this is exactly the function served by grid cells in the entorhinal cortex, and I used that as my starting point.

Third, in regards to representing space, I'm skeptical that we maintain an internal allocentric coordinate system that privileges a specific point as the origin... so to me, if there are these "spatial keys" that represent space, it can't be the case that we require the underlying Euclidean coordinates to produce these keys, the keys must be able to be produced RELATIVE to each other.

The end result of pondering and investigating these questions was, I discovered later, a sort of variation on the method of Random Fourier Features.

First, let's imagine there's a space we want to represent, \(\mathbb{R}^d\). We're going to represent \(\mathbb{R}^d\) in terms of $N >> d$ complex plane waves, which are conceptually analogous to grid cells (actually, they are a VERY close analog to BAND cells). Let's say that \(\mathcal{C} = \{z \in \mathbb{C} : |z|=1\}\) is the set of all unit-magnitude complex numbers. Then, each complex plane wave is a function \(k_j : \mathbb{R}^d \rightarrow \mathcal{C}\). Each complex plane wave \(k_j\) is going to have an ORIENTED FREQUENCY, given by the vector \(\newcommand{\vect}[1]{\boldsymbol{#1}}\vect{\gamma}_j \in \mathbb{R}^d\), where the direction of \(\vect{\gamma}_j\) specifies the direction in which the plane wave is pointing, and its magnitude specifies the frequency of the wave. This means an individual plane wave has this form: \[k_j(x) = k_j(\vect{0})e^{2\pi i \vect{\gamma}_j^{\top}\vect{x}}\] where \(k_j(\vect{0})\) is a phase offset, the value of \(k_j\) at the origin \(\vect{0} \in \mathbb{R}^d\).

Now suppose we have \(N\) of these complex plane waves, we could organize the value of all of these plane waves into a vector \[\vect{k}(x) = \vect{k}(\vect{0}) \odot e^{2\pi i \mathbf{\Gamma} \vect{x}}\]where \(\mathbf{\Gamma}\) is just a matrix composed of all the corresponding oriented frequency vectors \(\gamma_j\), which we assume are each drawn from some distribution \(p_{\vect{\gamma}}(\vect{\gamma}) : \mathbb{R}^d \rightarrow \mathbb{R}\).

Now, \(\vect{k}(\vect{x}) : \mathbb{R}^d \rightarrow \mathcal{C}^N\) is a function that assigns points in $\mathbb{R}^d$ to complex-valued "key" vectors in $|mathcal{C}^N... this is fine, but it requires us to have access to the underlying Euclidean coordinate space. Let's fix that. Consider the following, where \(\vect{x}' = \vect{x} + \vect{\delta}\):

\begin{align}
\vect{k}(\vect{x}') &= \vect{k}(\vect{0}) \odot e^{2\pi i \mathbf{\Gamma} \vect{x}'} \\
\vect{k}(\vect{x} + \vect{\delta}) &= \vect{k}(\vect{0}) \odot e^{2\pi i \mathbf{\Gamma} (\vect{x} + \vect{\delta})} \\
 &= \vect{k}(\vect{0}) \odot e^{2\pi i \mathbf{\Gamma} \vect{x} + 2\pi i \mathbf{\Gamma} \vect{\delta}} \\
 &= \vect{k}(\vect{0}) \odot e^{2\pi i \mathbf{\Gamma} \vect{x}} \odot e^{2\pi i \mathbf{\Gamma} \vect{\delta}} \\
 &= \vect{k}(\vect{x}) \odot e^{2\pi i \mathbf{\Gamma} \vect{\delta}}
\end{align}This means that if we have an existing key-vector, we can define OTHER key vectors RELATIVE to it... we can actually discard the underlying Euclidean coordinate system! Or at least, we can discard the choice of a specific ORIGIN, we still have to use RELATIVE coordinates via the displacement vector \(\vect{\delta} \in \mathbb{R}^d\).

Well, what does this actually buy us? To understand that, we have to remember what the point of this was, which was to generate spatially-specific "key" vectors so that we could store "values" in a kind of "dictionary", via a heteroassociative memory. Let's look at that memory system real quick, it's very simple.

Let's define a "bind" operation, \[\newcommand{\bind}{\triangleright}\newcommand{\norm}[1]{||#1||}\newcommand{\cossim}[2]{\langle\!\langle#1, #2\rangle\!\rangle}\vect{k} \bind \vect{v} = \frac{\vect{v}\vect{k}^*}{\norm{\vect{k}}^2}\]which is just a scaled outer product of the complex-valued vectors \(\vect{k}\) and \(\vect{v}\), which has the property that \[(\vect{k} \bind \vect{v}) \vect{q} = \vect{v}\frac{\norm{\vect{q}}}{\norm{\vect{k}}}\cossim{\vect{k}}{\vect{q}}\]where \(\cossim{\vect{k}}{\vect{q}}\) is the cosine-similarity between \(\vect{k}\) and \(\vect{q}\). What's important is this consequence:\[\left[\displaystyle\sum_{i=1}^{n}\vect{k}_i \bind \vect{v}_i\right]\vect{q} = \sum_{i=1}^{n}\vect{v}_i\frac{\norm{\vect{q}}}{\norm{\vect{k}_i}}\cossim{\vect{k}_i}{\vect{q}}\]Each "binding", \(\vect{k}_i \bind \vect{v}_i\), is a matrix, so the what this expression says is that if we take a sum of these matrices, the result of querying the matrix is equal to the sum of all the "value" vectors, weighted by the cosine-similarity of their corresponding "key" vectors to the "query" vector \(\vect{q}\).

Now, the vectors generated by the function \(\vect{k}(\vect{x} + \vect{\delta}) = \vect{k}(\vect{x}) \odot e^{2\pi i \mathbf{\Gamma} \vect{\delta}}\) are going to act like "keys" for our matrix dictionary, though, we're going to do a little element-wise modulation, you'll see why soon. Rather than storing and querying an association like this: \[\vect{v}' = [\vect{k}(\vect{x} \bind \vect{v})] \vect{q}\], we'll instead store and query an association like this: \[\vect{v}' = [(\vect{\mu} \odot \vect{k}) \bind \vect{v}](\vect{\eta} \odot \vect{q})\]

Let's look at what happens when we store a value \(\vect{v}\) at the key \((\vect{\mu} \odot \vect{k}(\vect{x}))\), and then query at \((\vect{\eta} \odot \vect{k}(\vect{x} + \vect{\delta}))\):\[\begin{align}
\vect{v}' &= [(\vect{\mu} \odot \vect{k}(\vect{x})) \bind \vect{v}](\vect{\eta} \odot \vect{k}(\vect{x} + \vect{\delta})) \\
&= \vect{v}\frac{\norm{\vect{\eta} \odot \vect{k}(\vect{x} + \vect{\delta})}}{\norm{\vect{\mu} \odot \vect{k}(\vect{x})}}\cossim{\vect{\mu} \odot \vect{k}(\vect{x})}{\vect{\eta} \odot \vect{k}(\vect{x} + \vect{\delta})} \\
&= \vect{v}\frac{\norm{\vect{\eta}}}{\norm{\vect{\mu}}}\frac{(\vect{\mu} \odot \vect{k}(\vect{x}))^*(\vect{\eta} \odot \vect{k}(\vect{x} + \vect{\delta}))}{\norm{\vect{\mu}} \cdot \norm{\vect{\eta}}} \\
&= \vect{v}\frac{1}{\norm{\vect{\mu}}^2}\sum_{j=1}^K{\overline{\mu}_j\overline{k}_j(\vect{x})\eta_jk_j(\vect{x} + \vect{\delta})} \\
&= \vect{v}\frac{1}{\norm{\vect{\mu}}^2}\sum_{j=1}^K{\overline{\mu}_j\overline{k}_j(\vect{x})\eta_jk_j(\vect{x})e^{2\pi i \vect{\gamma}^{\top}\vect{\delta}}} \\
&= \vect{v}\frac{1}{\norm{\vect{\mu}}^2}\sum_{j=1}^K{\overline{\mu}_j \eta_j e^{2\pi i \vect{\gamma}_j^{\top}\vect{\delta}}}
\end{align}\]

If you squint at this term \(\sum_{j=1}^K{\overline{\mu}_j \eta_j e^{2\pi i \vect{\gamma}_j^{\top}\vect{\delta}}}\), you might recognize the Fourier transform. Let's express the individual elements of \(\vect{\mu}\) and \(\vect{\eta}\) as functions of \(\vect{\gamma}_j\), so basically:\[\begin{align}
\mu_j &= \mu(\vect{\gamma}_j) \\
\eta_j &= \eta(\vect{\gamma}_j)
\end{align}\]If we then let \(g_j = g(\vect{\gamma}_j = \overline{\mu}(\vect{\gamma}_j) \cdot \eta(\vect{\gamma}_j)\), then we can actually derive the following:\[g(\vect{\gamma}_j) = \frac{\hat{c}(\vect{\gamma}_j)}{p_{\vect{\gamma}}(\vect{\gamma}_j)}\]which means that the "weights" \(\overline{mu}(\vect{\gamma}_j) \cdot \eta(\vect{\gamma}_j)\) are equal to the Fourier transform of the "response function" \(c(\vect{\delta})\) divided by the probability density of the oriented frequencies. This lets us actually CHOOSE \(c(\vect{\delta})\) and DERIVE the required weights to get that response function!
